Predicting Titled Tuesday Game Outcomes (White's perspective)¶

Task. For each Titled Tuesday blitz game, predict White's result — white win / draw / white loss — using only information available before the game starts. Leakage-safe data collection and feature engineering live in src/; this notebook is the modeling + evaluation deliverable and reuses them.

Approach. Outcomes are ordinal (a draw sits between a loss and a win), so instead of a flat 3-class model we use the cumulative / Frank–Hall decomposition: two binary XGBoost classifiers recombined into a proper 3-class distribution. Model selection uses leave-one-event-out cross-validation to (1) pick a parsimonious feature subset and (2) tune the XGBoost hyper-parameters; we evaluate once on a held-out future event and benchmark against the natural baseline — the higher-rated player wins.

Headline finding. Chess.com's game-object rating columns report each player's rating after the match, not before it. The pipeline reconstructs true pre-game ratings from each player's most recent earlier game and excludes raw white_rating / black_rating from modeling. With that correction, the honest model lands right on the baseline (~0.70 accuracy): pre-game blitz outcomes are mostly determined by ratings, and the ordinal model's value is calibrated probabilities rather than a better argmax.

Feature set & the temporal split¶

Split. A temporal (out-of-time) split: predict the future from the past, exactly how the model would run live, and the only split that does not leak tournament-level information (pairings, who showed up) across the boundary. Three earlier weeks are history-only context that populate the form / head-to-head / pre-game-rating features but contribute no rows.

split events role
history Jan 20 / 27 · Feb 03 feature context only (no rows)
train Feb 10 / 17 / 24 fit + hyper-parameter tuning
validation Mar 03 model selection + reported metrics
test Mar 10 touched once, final verdict

Three training events (not one) let model selection cross-validate by event (leave-one-week-out), so a config must generalize to an unseen week.

Features. All inputs are known before the first move. We use the reconstructed pre-game ratings (white_pregame_rating / black_pregame_rating and everything derived from them) and exclude the raw post-game white_rating / black_rating from the game object because Chess.com stores those ratings after the match. We also drop identifiers, the target, and the collinear black_expected_score (= 1 − white_expected_score).

Feature glossary. Every model feature is white-perspective and can be explained in one sentence:

Modeling utilities¶

OrdinalXGB wraps two cumulative XGBoost heads — P(y≥1) ("White does not lose") and P(y≥2) ("White wins") — and recombines them into a 3-class distribution. Each head is class-balanced with scale_pos_weight, and passing a held-out event turns on early stopping. Independent heads can be non-monotone — occasionally P(y≥2) > P(y≥1), which would make the recovered draw probability negative — so we clip and renormalize, and later report how often it happens.

def recover_proba(p_ge_draw, p_ge_win):
    # Recombine cumulative heads into a proper 3-class distribution.
    p_black = 1.0 - p_ge_draw
    p_draw = p_ge_draw - p_ge_win           # < 0 if the heads disagree
    p_white = p_ge_win
    violation = p_ge_win > p_ge_draw
    P = np.clip(np.vstack([p_black, p_draw, p_white]).T, 0.0, None)
    P = P / P.sum(axis=1, keepdims=True)
    return P, violation


class OrdinalXGB:
    # Frank-Hall ordinal classifier: two XGB heads for P(y>=1) and P(y>=2).
    # Each head is class-balanced with scale_pos_weight; passing `eval_ord`
    # turns on early stopping against a held-out event. Fit on a DataFrame so
    # the booster keeps real feature names (needed for gain importance).

    def __init__(self, **params):
        self.params = params

    def fit(self, X, y_ord, eval_ord=None):
        y_ord = np.asarray(y_ord)
        heads = {"ge_draw": (y_ord >= 1).astype(int),
                 "ge_win": (y_ord >= 2).astype(int)}
        self.models_ = {}
        for name, y_head in heads.items():
            pos = max(int(y_head.sum()), 1)
            kw = dict(self.params, scale_pos_weight=int((1 - y_head).sum()) / pos)
            if eval_ord is not None:
                Xe, ye = eval_ord
                thr = 1 if name == "ge_draw" else 2
                kw["early_stopping_rounds"] = 30
                model = XGBClassifier(**kw)
                model.fit(X, y_head,
                          eval_set=[(Xe, (np.asarray(ye) >= thr).astype(int))],
                          verbose=False)
            else:
                model = XGBClassifier(**kw).fit(X, y_head)
            self.models_[name] = model
        return self

    def predict_proba(self, X):
        p_d = self.models_["ge_draw"].predict_proba(X)[:, 1]
        p_w = self.models_["ge_win"].predict_proba(X)[:, 1]
        P, violation = recover_proba(p_d, p_w)
        self.violation_rate_ = float(violation.mean())
        return P

    def predict(self, X):
        return self.predict_proba(X).argmax(axis=1)

def summarize(name, y_true, y_pred, proba=None):
    row = {"model": name,
           "accuracy": accuracy_score(y_true, y_pred),
           "macro_F1": f1_score(y_true, y_pred, average="macro"),
           "draw_F1": f1_score(y_true, y_pred, labels=[1], average="macro")}

        row["macro_AUC"] = roc_auc_score(yb, proba, average="macro",
                                         multi_class="ovr")
    return row

2. Rating & Elo EDA¶

Ratings are the dominant pre-game signal. We use each player's reconstructed pre-game rating; the raw game-object rating is unsafe because it is recorded after the match. Two ratings become one expectation via the Elo expected-score formula (in src/features/rating.py):

$$E_{\text{white}} = \frac{1}{1 + 10^{\,(R_{\text{black}} - R_{\text{white}})/400}}$$

$E_{\text{white}}\in(0,1)$ is White's expected score (win=1, draw=½, loss=0), not a win probability — draws pull realized scores toward ½. Derived features: rating_diff = R_white − R_black, abs_rating_diff, avg_rating, white_expected_score — all on the pre-game ratings.

Rating leakage note¶

The raw Chess.com game-object rating columns (white_rating / black_rating) are post-match values: they report each player's rating after this game, not the rating available before move one. The model therefore uses reconstructed pre-game ratings from each player's most recent strictly earlier game and excludes those raw rating columns from FEATURES. Everything below uses that clean, pre-game feature set.

3. Cold-start analysis¶

The next cell checks rating debuts, where a player has no strictly earlier game in the collected window. "Cold-start" also covers games where the form / head-to-head families have little history. The feature layer zero-fills these, so we detect them via indicators:

  • In-tournament cold: round == 1 (no earlier round this event).
  • Head-to-head cold: h2h_games == 0 (these two never met).
  • Cross-event cold: no prior-week history (both form diffs at 0).

5. Cross-validated model selection¶

We keep the XGBoost + Frank–Hall ordinal model and push it in two simple, leakage-safe steps, both scored by leave-one-event-out GroupKFold (groups = training event) so every choice must generalize to an unseen week:

  1. Feature selection. Rank features by XGBoost gain averaged across the CV folds, then sweep top-k subsets and keep the smallest set whose CV macro-F1 is within one standard deviation of the best — a parsimonious cut that drops noise features without hurting the score.
  2. Hyper-parameter tuning. A randomized search over a richer XGBoost grid (depth, learning rate, trees, min_child_weight, subsampling, gamma, reg_lambda) on the selected features.

The number to beat is the higher-rated-wins baseline scored on the same folds, so the comparison is apples-to-apples.

from sklearn.model_selection import GroupKFold

gkf = GroupKFold(n_splits=3)
folds = list(gkf.split(X_train, y_train, groups_train))

# The higher-rated-wins baseline on the same folds -- the number to beat.
base_cv_f1 = float(np.mean([
    f1_score(y_train[va], baseline_predict(tr.iloc[va]), average="macro")
    for _, va in folds]))


def cv_macro_f1(features, params):
    # Leave-one-event-out CV macro-F1 for an OrdinalXGB fit on `features`.
    scores = []
    for tr_idx, va_idx in folds:
        m = OrdinalXGB(**params).fit(X_train.iloc[tr_idx][features], y_train[tr_idx])
        scores.append(f1_score(y_train[va_idx],
                               m.predict(X_train.iloc[va_idx][features]),
                               average="macro"))
    return float(np.mean(scores)), float(np.std(scores))


# --- Step 1: rank features by CV-averaged XGBoost gain, then sweep top-k. ---
SELECT_PARAMS = dict(BASE_PARAMS, max_depth=4, learning_rate=0.1,
                     n_estimators=300, subsample=0.8, colsample_bytree=0.8)

gain = pd.Series(0.0, index=FEATURES)
for tr_idx, _ in folds:
    m = OrdinalXGB(**SELECT_PARAMS).fit(X_train.iloc[tr_idx], y_train[tr_idx])
    for head in m.models_.values():
        for f, v in head.get_booster().get_score(importance_type="gain").items():
            gain[f] += v
ranking = gain.sort_values(ascending=False)

sweep = []
for k in [2, 3, 5, 8, 12, len(FEATURES)]:
    f1, sd = cv_macro_f1(ranking.index[:k].tolist(), SELECT_PARAMS)
    sweep.append({"k": k, "cv_macro_F1": f1, "cv_std": sd})
sweep = pd.DataFrame(sweep)

tol = best["cv_macro_F1"] - best["cv_std"]

def permutation_importance_f1(model, X, y, n_repeats=5, seed=RNG):
    rng = np.random.default_rng(seed)
    base_f1 = f1_score(y, model.predict(X), average="macro")
    out = {}
    for col in X.columns:
        drops = []
        for _ in range(n_repeats):
            Xp = X.copy()
            Xp[col] = rng.permutation(Xp[col].to_numpy())
            drops.append(base_f1 - f1_score(y, model.predict(Xp), average="macro"))
        out[col] = float(np.mean(drops))
    return pd.Series(out).sort_values(ascending=False)

9d. If the draw class is the goal: trade overall accuracy for draw recall¶

The main model — like the baseline — almost never predicts a draw, so its draw precision / recall / F1 are ~0. That is the right call only if the product needs a decisive label. If instead the goal is to flag likely draws, we can push the minority class with oversampling, stratified-by-class sampling, or a class-weighted loss. Below, a plain 3-class XGB (same tuned params and selected features) is trained unweighted and again with balanced class weights (compute_sample_weight) — the weighted-loss lever; oversampling the draw rows has the same effect. Same features, only the class emphasis changes, so the table isolates that one knob.

Result. Balancing makes the model actually call draws (287 test games) — draw precision / recall / F1 rise from 0 to ≈ 0.09 / 0.17 / 0.12 and macro-F1 edges up (0.48 → 0.50) — but overall accuracy drops 0.70 → 0.63. Draw AUC stays near chance (~0.53), so we can call more draws yet still can't rank them well: the pre-game draw signal is genuinely thin. Whether this is an improvement is a product decision — it depends entirely on whether draw calls are worth the accuracy hit.

10. Writeup & next steps¶

How I split the data and why¶

A temporal (out-of-time) split: three consecutive weeks to train (Feb 10 / 17 / 24), the next week to validate (Mar 03), the final week to test (Mar 10); three earlier weeks are history-only context feeding the form / head-to-head / pre-game-rating features. This mirrors real use — predict this week from past weeks — and avoids the leakage a random split invites, where near-identical games from one tournament (shared pairings, the same field, the same day's ratings) would straddle the split and inflate the score. Three training events let selection cross-validate by event (leave-one-week-out), so a chosen config must generalize to an unseen week. The test event is touched exactly once.

Is it a good model?¶

Honest verdict: it is a correct, leak-free model that essentially ties the simple baseline as a classifier, and adds value only as a probability model. Exact numbers are in the scoreboard above; the shape is stable:

  • One important data correction is that Chess.com's raw game-object rating columns are post-match values. The pipeline reconstructs true pre-game ratings from strictly earlier games and excludes the raw rating columns, keeping the model aligned with what would be known before move one.
  • On argmax accuracy and macro-F1 the model sits right on the "higher-rated wins" baseline (~0.70 / ~0.48 — an edge of ~0.001 accuracy, well within noise). Once leakage is gone, a scalar rating already captures almost all of the pre-game signal, so a gradient-boosted model with form / matchup / H2H features does not meaningfully beat the one-line rule.
  • Its genuine advantage is probabilistic: it outputs calibrated win/draw/loss probabilities with real discrimination on the decisive classes (ROC AUC ≈ 0.78) that the baseline cannot — useful anywhere a score, not just a label, is needed.
  • Draws are effectively unpredictable pre-game. Draw AUC ≈ 0.47 (below chance) and the model never calls a draw. There is almost no pre-game signal that specifically predicts a draw; the thin draw band shows up as the ~17% monotonicity-violation rate. This is largely irreducible in blitz.
  • "Good" depends on the goal — and the goal fixes the metric. For a decisive label, the baseline is the sensible choice: near-identical results with far less complexity. For draw detection, a model earns its place — up-weighting the draw class (§9d: oversampling / stratified-by-class sampling / weighted loss) lifts draw precision/recall/F1 from 0 to ≈ 0.09 / 0.17 / 0.12 and macro-F1 to ≈ 0.50, at the cost of accuracy (0.70 → 0.63). For calibrated probabilities, judge on log-loss / calibration and AUC, not argmax. So the metric follows the use case: broadcast overlays → probabilities, upset alerts → ranking / AUC, a raw call → accuracy (then handle the draw imbalance).
  • The ≈ 0.70 accuracy and ~7% draw rate are about what to expect for blitz between similar-Elo titled players: the slices show accuracy is high only when ratings are far apart (0–50 Elo ≈ 0.57 → 400+ ≈ 0.91) and slides toward a coin-flip in the later rounds as the field converges — no fragile regime, just the physics of Elo.

What I'd do next with more time¶

Conclusions & next steps. Before more modeling, pin down the goal and product use case — that choice fixes the evaluation metric, the preferred model, and the selection criterion. If raw decisive-outcome prediction is the goal, tackle class imbalance explicitly in training and validation; if calibrated probabilities are the goal, judge on log-loss / calibration rather than argmax accuracy. Talk to stakeholders and domain experts to confirm what problem is actually worth solving and whether a pre-game outcome predictor is even the right tool — versus better features, a different model, or a different approach entirely — then share with the team and gather ideas and prior learnings before investing further.

Possible product usages. These earn their keep only if the model gains a real probabilistic edge over raw rating (through calibration and more data); today it merely ties the baseline. With that edge:

  • Explainability of the most-probable outcome (per-game feature attribution).
  • Pre-game matchup preview.
  • Fairer pairing / matchmaking-quality checks.
  • Pre-game win/draw/loss probabilities for broadcast overlays and a "closest games right now" ranking; upset alerts where the model's probability diverges from raw rating; draw-likelihood to flag low-excitement pairings.
  • Personalized pre-game preparation.

Productionization. Sketch how this actually runs: a scheduled ETL that pulls each new Titled Tuesday event from the PubAPI, reconstructs pre-game ratings, and refreshes data/processed/ on each new event; a model-retraining cadence when warranted (rolling-origin refit as new weeks arrive, keeping the leakage guard and temporal split intact); and serving the pre-game probabilities behind a small inference interface.

Data improvements. Most important: a bigger dataset — many more events, ideally a full season — so form, head-to-head, and draw signals leave cold-start and become learnable; this is the single highest-value change. The earlier ideas still hold: time-control-specific historical ratings and rating volatility (added carefully, as-of-game snapshots only). Feature-engineering brainstorm: each player's historical openings and the opening compatibility between the two players' repertoires — which style/opening matchups tend toward decisive versus drawn games.
